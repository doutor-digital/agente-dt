Quickstart

Do zero ao webhook funcionando em ~5 minutos.

  1. 1

    Pré-requisitos

    • Node.js ≥ 22 (testado em 24)
    • pnpm ≥ 9 — corepack enable ou npm i -g pnpm
    • PostgreSQL ≥ 15 — local ou Docker
    • Kommo Long-Lived Access Token
    • Anthropic API Key
  2. 2

    Banco de dados

    Suba um Postgres local via Docker:

    docker run --name agente-dt-pg \
  -e POSTGRES_PASSWORD=postgres \
  -p 5432:5432 -d postgres:16

docker exec -it agente-dt-pg \
  psql -U postgres -c "CREATE DATABASE agente_dt;"
  3. 3

    Instalar e configurar

    Uma única instalação cobre backend e frontend (workspace pnpm):

    pnpm install                       # instala tudo
cp backend/.env.example backend/.env  # preencha KOMMO_* e ANTHROPIC_API_KEY
pnpm prisma:migrate                # cria tabelas no banco
    ℹ️ O postinstall do backend roda prisma generate automaticamente.
  4. 4

    Subir em dev

    pnpm dev   # roda backend (3001) + frontend (5173) em paralelo

    Abra localhost:5173 para o dashboard.

  5. 5

    Testar webhook

    Dispare um payload de teste sem precisar de Kommo configurado:

    curl -X POST http://localhost:3001/api/webhooks/kommo \
  -H "Content-Type: application/json" \
  -d '{"leadId": 12345, "message": "Tem orçamento aprovado, quer fechar."}'

    A execução aparece na sidebar do dashboard e o console renderiza cada passo em tempo real.

Webhook Controller

src/controllers/webhook.controller.ts — borda HTTP do sistema.

Regra crítica: a Kommo timeoutsa em 30 s. Nunca processe o agente sincronamente no handler.

Padrão: ACK rápido + processamento em background.

  1. 1 Valida payload com Zod. Aceita formato Kommo (leads[add][].id) e formato de teste (leadId).
  2. 2 Cria ExecutionTrace com status RUNNING e grava o primeiro step (WEBHOOK_RECEIVED).
  3. 3 Responde 200 com {traceId} em < 100 ms.
  4. 4 Dispara agente em background via void processAgent(...).catch(...). 
// Resumo da lógica (webhook.controller.ts)
export async function handleKommoWebhook(req, res) {
  const requestStart = performance.now(); // monotônico — ideal pra medir duração

  const parsed = webhookSchema.safeParse(req.body);
  if (!parsed.success) return res.status(400).json({ ok: false });

  const leadId = extractLeadId(parsed.data);
  const trace  = await prisma.executionTrace.create({
    data: { threadId: `lead-${leadId}`, leadId: String(leadId),
            input: req.body, status: 'RUNNING' },
  });

  res.status(200).json({ ok: true, traceId: trace.id }); // ACK imediato

  // background — não bloqueia o response
  void processAgent({ leadId, traceId: trace.id, requestStart, /*...*/ })
    .catch(err => logger.error({ err }, 'erro fatal no agente'));
}

src/agent/graph.ts — coração da IA. Loop ReAct clássico.

Como o State transita

                    └────────┬───────┘
                             │  State inicial:
                             │  { leadId, messages:[HumanMessage], traceId }

                    ┌────────────────┐

                    └────────┬───────┘
                             │
                             ▼

thread_id
Chave da conversa contínua. Usamos lead-{leadId} — mesmo lead, mesma thread, agente lembra do contexto anterior.
trace_id
Chave da execução pontual (uma invocação do grafo). É o que aparece no dashboard.

Checkpointer

A cada transição de nó, o PostgresSaver serializa o State inteiro e grava nas tabelas checkpoints, checkpoint_writes e checkpoint_blobs. Essas tabelas são criadas pela própria lib no boot via checkpointer.setup() — não passe migrate do Prisma nelas. 

// Singleton no boot do servidor
const checkpointer = PostgresSaver.fromConnString(env.DATABASE_URL);
await checkpointer.setup();   // cria tabelas se não existirem

const compiled = workflow.compile({ checkpointer });

// Invocação com thread_id = memória contínua
await compiled.invoke(
  { leadId, traceId, messages: [new HumanMessage("...")] },
  {
    configurable: { thread_id: `lead-${leadId}` },
    recursionLimit: 10,   // trava contra loop infinito de tool_calls
  },
);

Tools

src/agent/tools.ts — interface entre a LLM e o mundo real.

Cada tool tem três componentes que viram contrato com o Claude:

  • Schema Zod — vira o JSON Schema enviado à Messages API.
  • Descrição — texto que a LLM lê pra decidir quando chamar.
  • Função — executa e devolve string pra próxima iteração do ReAct.

const aplicar_tag = new DynamicStructuredTool({
  name: 'aplicar_tag',
  description: 'Adiciona uma tag ao lead no Kommo. Use quando a análise...',
  schema: z.object({
    leadId: z.number().int().positive(),
    tag:    z.string().min(1).max(50),
  }),
  func: async ({ leadId, tag }) => {
    await recorder.step({ kind: 'TOOL_CALL', title: `Aplicar tag "${tag}"`, /*...*/ });
    try {
      await KommoService.addTag({ leadId, tag });     // ← service puro
      await recorder.step({ kind: 'KOMMO_ACTION', /*...*/ });
      return `OK — tag "${tag}" aplicada.`;
    } catch (err) {
      await recorder.step({ kind: 'ERROR', /*...*/ });
      return `ERRO: ${err.message}`;  // não throw — deixa a LLM reagir
    }
  },
});

Kommo Service

src/services/kommo.service.ts — camada HTTP pura.

Essa camada é deliberadamente burra: traduz chamadas tipadas em requisições HTTP. Nada mais. Não conhece LangGraph, não conhece Prisma, não conhece logger de aplicação além do necessário pra diagnóstico. Se amanhã trocarmos Kommo por HubSpot, só esse arquivo muda.

Persistência

prisma/schema.prisma — duas responsabilidades distintas no mesmo banco.

Owner: nós

ExecutionTrace + ExecutionStep

Tabelas de observabilidade. Cada webhook vira um Trace, cada passo do grafo vira um Step. Gerenciadas via prisma migrate.

Owner: LangGraph

checkpoints, checkpoint_writes, checkpoint_blobs

Memória binária do agente. Criadas pelo PostgresSaver.setup() no boot. Não rodar migrate nelas.

Por que separar Trace de Step?

  • Trace é o envelope da execução: leadId, latency_ms, status.
  • Step é o raciocínio: cada chamada à LLM, cada tool call, cada ação no Kommo.
  • O dashboard renderiza um Step por linha do feed vertical.

Por que não só o checkpoint do LangGraph?

O checkpoint é otimizado para retomada de execução — serializado binário, não navegável. Para UI/observabilidade humana, tabela própria. Os dois coexistem.

Observabilidade

src/agent/trace-recorder.ts — ponte entre o grafo e o banco.

O LangGraph não conhece nosso domínio de observabilidade — nem deve. O TraceRecorder é o adapter que cada nó/tool chama pra gravar um ExecutionStep. Falhas de gravação nunca derrubam o agente — logam e seguem. 

await recorder.step({
  kind: 'TOOL_CALL',
  title: `Decisão: aplicar tag "${tag}" no lead ${leadId}`,
  payload: { leadId, tag },
  latencyMs: 312,   // duração desse passo específico
});

// Ao final da execução
await recorder.finalize({
  status: 'SUCCESS',
  latencyMs: totalLatency,
  iaDecision: result.decision,
});

Logs estruturados

Em produção, pino escreve JSON com threadId, leadId e traceId em todo log — basta um filtro no Loki/Datadog pra puxar uma execução inteira.

Troubleshooting

ERR_PNPM_NO_MATCHING_VERSION no install
Alguma versão pinada no package.json não existe no registry. Rode pnpm view <pkg> version pra ver a latest e ajuste.
Prisma reclama datasource property url is no longer supported
Você instalou Prisma 7 — esse MVP usa Prisma 6. Volte para ^6.19.3 ou migre o schema pra prisma.config.ts + adapter.
Webhook responde 200 mas nada aparece no dashboard
Cheque os logs do backend (LOG_LEVEL=debug). Causa comum: ANTHROPIC_API_KEY inválida — o agente falha mas o trace foi criado como FAILED. Olhe na tabela execution_traces.errorMessage.
Kommo retenta o mesmo webhook várias vezes
Significa que você respondeu em > 30s ou com status != 200. Verifique o handler webhook.controller.ts — o res.status(200) tem que vir antes de qualquer chamada ao agente.
Como ver o State serializado de uma execução?
Use pnpm prisma:studio e abra a tabela checkpoints filtrando por thread_id = lead-{leadId}. O conteúdo está em bytes (msgpack) — pode usar a lib pra deserializar se precisar inspecionar a fundo.
LANGGRAPH

Referência LangGraph.js

LangGraph é o framework de orquestração de agentes que dirige o agente deste projeto. Esta seção é uma referência condensada da doc oficial em docs.langchain.com/oss/javascript/langgraph — com foco no que é usado em backend/src/agent/*.

O que é

Framework de baixo nível para construir e executar agentes stateful, com persistência em checkpoints, retomada após falha e human-in-the-loop. Diferente de wrappers de alto nível, o LangGraph é puro grafo de estados — você define nodes (funções) e edges (transições, fixas ou condicionais).

  • Durable Execution — checkpoints automáticos a cada passo.
  • Human-in-the-Loop — inspecionar e modificar o State em qualquer ponto.
  • Memória — curta (mensagens) + longa (Store).
  • LangSmith — observabilidade integrada.

Instalação

pnpm add @langchain/langgraph @langchain/core
pnpm add @langchain/anthropic                        # provider Claude
pnpm add @langchain/langgraph-checkpoint-postgres    # persistência

State & Annotation

O State é o objeto que transita entre os nós do grafo. Cada nó recebe o State atual e retorna um delta (atualização parcial). O LangGraph aplica os reducers definidos em cada campo do Annotation.Root para mesclar o delta no State.

Definição com Annotation.Root

import { Annotation, messagesStateReducer } from '@langchain/langgraph';
import type { BaseMessage } from '@langchain/core/messages';

export const AgentState = Annotation.Root({
  // Reducer canônico para mensagens — faz concat respeitando IDs (upsert
  // pra streaming) e mantém ordem cronológica.
  messages: Annotation<BaseMessage[]>({
    reducer: messagesStateReducer,
    default: () => [],
  }),

  // "Last value wins": cada update sobrescreve.
  leadId: Annotation<number>({
    reducer: (_prev, next) => next,
    default: () => 0,
  }),

  // Reducer customizado — acumula em array.
  steps: Annotation<string[]>({
    reducer: (prev, next) => [...prev, ...next],
    default: () => [],
  }),
});

export type AgentStateType = typeof AgentState.State;

Reducers — semântica

Default (sem reducer)

Último valor vence — cada update sobrescreve o anterior.

messagesStateReducer

Concat com upsert por id. Use para histórico de chat — é o canal do loop ReAct.

Custom (prev, next) => ...

Você decide. Útil para listas append-only, somas, ou merge profundo.

MessagesAnnotation

Atalho pronto para grafos cujo único campo é messages.

Nodes & Edges

Nodes são funções (sync ou async) que recebem o State e devolvem um delta. Edges conectam nodes — podem ser fixas (addEdge) ou condicionais (addConditionalEdges). Os marcadores START e END são pseudo-nós para entrada e saída.

Construção do grafo

import { END, START, StateGraph } from '@langchain/langgraph';

const agentNode = async (state: AgentStateType) => {
  const response = await model.invoke(state.messages);
  return { messages: [response] };           // delta — o reducer concatena
};

const toolNode = new ToolNode(tools);        // helper do prebuilt

// Edge condicional: inspeciona última mensagem pra decidir a rota.
const shouldContinue = (state: AgentStateType): 'tools' | typeof END => {
  const last = state.messages.at(-1) as AIMessage | undefined;
  return last?.tool_calls?.length ? 'tools' : END;
};

const workflow = new StateGraph(AgentState)
  .addNode('agent', agentNode)
  .addNode('tools', toolNode)
  .addEdge(START, 'agent')
  .addConditionalEdges('agent', shouldContinue, {
    tools: 'tools',
    [END]: END,
  })
  .addEdge('tools', 'agent');                // loop ReAct

const compiled = workflow.compile({ checkpointer });
Loop ReAct. A aresta tools → agent fecha o loop clássico: o LLM pede tool → tool executa → resultado vira ToolMessage → LLM decide o próximo passo. A trava é recursionLimit no invoke (default 25).

Tools & ToolNode

Tools são a interface entre o LLM e o mundo real. Cada tool tem um schema Zod (que vira o formato de tool-calling da Anthropic) e uma descrição em linguagem natural — é a descrição que faz o LLM decidir quando chamar a tool. ToolNode é o nó pronto que executa as tools chamadas pela última AIMessage e devolve ToolMessages.

Definir uma tool com Zod

import { DynamicStructuredTool } from '@langchain/core/tools';
import { z } from 'zod';

const aplicar_tag = new DynamicStructuredTool({
  name: 'aplicar_tag',
  // ESSA descrição é o "gatilho" — o LLM lê isso pra decidir quando chamar.
  description:
    'Adiciona uma tag ao lead no Kommo. Use quando a análise indicar uma ' +
    'classificação (ex: "Quente", "Frio"). Idempotente.',
  schema: z.object({
    leadId: z.number().int().positive().describe('ID numérico do lead.'),
    tag: z.string().min(1).max(50).describe('Nome da tag, ex: "Quente".'),
  }),
  func: async ({ leadId, tag }) => {
    await KommoService.addTag({ leadId, tag });
    return `OK — tag "${tag}" aplicada no lead ${leadId}.`;
  },
});

import { ChatAnthropic } from '@langchain/anthropic';

Regra de ouro. A descrição da tool é o seu prompt indireto. Descreva o caso de uso ("use quando..."), não a implementação ("isso chama um endpoint REST"). O LLM nunca vê seu código.

Erros que NÃO quebram o loop

Dentro de func, capture exceções e retorne uma string descrevendo a falha. Lançar quebra o loop ReAct; retornar deixa o LLM reagir (ex: tentar outra tool).

Persistência (Checkpointer)

Ao compilar um grafo com checkpointer, o LangGraph grava um snapshot do State a cada super-step. Snapshots são agrupados por thread_id — a chave da conversa. Próximas invocações com o mesmo thread_id rehidratam o State e o agente "lembra" do contexto anterior.

Setup do PostgresSaver

import { PostgresSaver } from '@langchain/langgraph-checkpoint-postgres';

const cp = PostgresSaver.fromConnString(process.env.DATABASE_URL!);
await cp.setup();           // cria as tabelas checkpoints/checkpoint_writes/blobs
const graph = workflow.compile({ checkpointer: cp });

// Cada invocação precisa de thread_id.
await graph.invoke(
  { messages: [{ role: 'user', content: 'Cliente fechou!' }] },
  { configurable: { thread_id: 'lead-12345' } },
);

Inspecionar State / histórico

const config = { configurable: { thread_id: 'lead-12345' } };

// Snapshot mais recente
const snap = await graph.getState(config);
//  → { values, next, config, metadata, createdAt, parentConfig }

// Todo o histórico (iterator)
for await (const s of graph.getStateHistory(config)) {
  console.log(s.metadata.step, s.values);
}

// Retomar a partir de um checkpoint específico
await graph.invoke(null, {
  configurable: {
    thread_id: 'lead-12345',
    checkpoint_id: '1ef663ba-28fe-6528-8002-5a559208592c',
  },
});

Streaming

Para feedback em tempo real (como o feed do dashboard deste projeto), use graph.stream(). Há vários stream modes — cada um emite um tipo de evento. Você pode passar um array pra combinar modos.

Modos disponíveis

Mode Emite Quando usar
valuesState completo após cada stepSnapshots por passo
updatesSó o delta de cada nodeLogs leves, audit trail
messagesTokens do LLM [chunk, metadata]Typewriter no front
toolsCiclo de vida das toolsProgresso de tool calls
customDados emitidos via config.writer()Eventos do seu negócio
debugTudo (verboso)Debug local

Streamando tokens do LLM

for await (const [chunk, metadata] of await graph.stream(
  { messages: [{ role: 'user', content: 'Olá' }] },
  { streamMode: 'messages', configurable: { thread_id: 'lead-1' } },
)) {
  // Filtra por nó se quiser só os tokens do node "agent"
  if (metadata.langgraph_node === 'agent' && chunk.content) {
    process.stdout.write(chunk.content);
  }
}

Combinando modos

for await (const [mode, chunk] of await graph.stream(input, {
  streamMode: ['updates', 'tools'],
})) {
  if (mode === 'updates') console.log('node updated:', chunk);
  if (mode === 'tools' && chunk.event === 'on_tool_end') {
    console.log('tool done:', chunk.name, chunk.output);
  }
}
Roadmap deste projeto. Hoje o dashboard usa polling de /api/traces/:id. O próximo passo é substituir por SSE emitindo eventos derivados de graph.stream({streamMode:['updates','tools']}) — pra eliminar a latência de 1s do polling.
